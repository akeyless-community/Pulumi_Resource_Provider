# Releasing the provider (v0.1.0 and later)

This document walks through **pushing code**, **tagging**, the **automated** upload of the six Pulumi provider `.tar.gz` files to GitHub Releases, and where to get **npm / PyPI / NuGet** tokens. After that, you can **fork [pulumi/registry](https://github.com/pulumi/registry)** and add this package to the list (see the main [README](README.md)).

## 0. Prereqs

- Git, GitHub access to [akeyless-community/pulumi-akeyless](https://github.com/akeyless-community/pulumi-akeyless).
- Optional: [GitHub CLI](https://cli.github.com/) (`gh`) to create releases and watch Actions from the terminal.

## 1. Prepare the release commit (version `0.1.0`)

The Go linker and generated SDKs use `PROVIDER_VERSION` — **no leading `v`**.

```bash
cd /path/to/pulumi-akeyless
make generate PROVIDER_VERSION=0.1.0
# or: make build PROVIDER_VERSION=0.1.0
```

Review the diff (e.g. `package.json`, `pyproject.toml`, `schema` outputs), then commit and push to `main` (or your release branch):

```bash
git add -A
git status
git commit -m "chore: release v0.1.0"
git push origin main
```

## 2. Create and push the Git tag

A **Git tag** marks the exact commit for the release. Pulumi and GitHub use the form **`v` + SemVer** (e.g. `v0.1.0`).

```bash
git tag v0.1.0
git push origin v0.1.0
```

Or with `gh` (if you use it to confirm the remote):

```bash
gh repo sync  # if needed
git push origin main
git tag v0.1.0
git push origin v0.1.0
```

## 3. What happens on GitHub (upload is automated)

Pushing the tag runs [.github/workflows/release.yml](.github/workflows/release.yml):

1. Derives `PROVIDER_VERSION` from the tag (e.g. `v0.1.0` → `0.1.0`).
2. Runs `make provider_dist` with `SKIP_SIGNING=true` (unsigned Windows binaries; fine for public use).
3. **Creates a GitHub Release** for that tag and **attaches the six** `pulumi-resource-akeyless-v0.1.0-*.tar.gz` files to it.

In the GitHub UI: **Actions** → **Release provider** — wait for a green run.  
Then **Releases** — you should see `v0.1.0` with six assets.

If the workflow fails, open the failed job log; common issues are Go version drift or tag format.

## 4. `GITHUB_TOKEN` and permissions

The workflow uses the default **`GITHUB_TOKEN`** (no extra secret). The workflow sets `permissions: contents: write` so the job can create the release and upload files. You only need a custom token if you change the workflow to publish somewhere else or use a stricter org policy; in that case, store a **PAT** with `contents: write` in a secret and use it in the release step (not required for the default setup).

## 5. Public SDKs: where tokens come from

Publishing `@pulumi/akeyless`, `pulumi_akeyless`, and `Pulumi.Akeyless` is **separate** from the GitHub Release. You run publishing locally or from another workflow **after** the provider version is final. Tokens are always created in the **registry’s website** for your org/user — nothing is “generated by GitHub” for npm/PyPI/NuGet except in advanced OIDC setups.

### npm (Node.js, package `@pulumi/akeyless`)

1. Log in at [https://www.npmjs.com/](https://www.npmjs.com/) (or your org, e.g. Pulumi’s scope if they grant access).
2. Profile → **Access Tokens** → **Generate New Token** → **Granular** or **Automation** (for CI) with permission to publish the relevant package.
3. Save the token once; set it in CI as `NPM_TOKEN` (or `NODE_AUTH_TOKEN` depending on the tool), or `npm config set` locally for one-off publish.

**Note:** Publishing under the **`@pulumi`** scope usually requires Pulumi to add your npm user to that scope. If you do not have that access, you would change the package name in bridge config, regenerate, and publish under your own scope (e.g. `@akeyless-community/pulumi-akeyless`).

### PyPI (Python, package `pulumi_akeyless`)

1. [https://pypi.org/manage/account/](https://pypi.org/manage/account/) → **API tokens**.
2. Create a token scoped to a project (once the project exists) or to the whole account (less ideal for production).
3. In CI, use the token with **Twine** or `uv publish` (often the env var is `PYPI_API_TOKEN` or `TWINE_PASSWORD` with user `__token__`); see [PyPI’s token help](https://pypi.org/help/#apitoken).

**Trusted publishing (optional):** PyPI can link a GitHub repo to publish without a long‑lived token; set that up in the PyPI project’s settings if you want OIDC from GitHub Actions later.

### NuGet (.NET, package `Pulumi.Akeyless`)

1. [https://www.nuget.org/account/apikeys](https://www.nuget.org/account/apikeys) — **Create** a key.
2. Set **Select scopes** to **Push** for `Pulumi.Akeyless` (or global push if you are the owner and accept the risk).
3. Copy the key once. In CI it is often `NUGET_API_KEY` and used with `dotnet nuget push`.

## 6. After assets and (optional) SDKs

- The Pulumi CLI will download the plugin from GitHub for users once **`pluginDownloadURL`** is set (already in this repo) and a matching **tag + assets** exist.
- To list the package on the public registry, fork **[pulumi/registry](https://github.com/pulumi/registry)**, add the `repoSlug` / `schemaFile` entry to `community-packages/package-list.json`, and open a PR (see [README](README.md)).

## 7. Manual release (if you skip the workflow)

If you do not use Actions, you can build locally and upload with the GitHub UI or `gh`:

```bash
export SKIP_SIGNING=true
make provider_dist PROVIDER_VERSION=0.1.0
# Then create a release for tag v0.1.0 and upload bin/*.tar.gz, or:
gh release create v0.1.0 bin/pulumi-resource-akeyless-v0.1.0-*.tar.gz --title v0.1.0 --generate-notes
```

Ensure the **tag** `v0.1.0` already exists on GitHub.
